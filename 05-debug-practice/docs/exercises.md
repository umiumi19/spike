# 演習問題

各演習のゴールは共通です：

1. まず普通に実行して、出てくる **panic（実行時エラー）** を自分の目で確認する。
2. ブレークポイントを置いてデバッグ実行し、**どの変数が、いつ、なぜ想定外の値になったのか**を突き止める。
3. 原因を一言で説明できるようにする（修正は `solutions.md` で確認）。

> 実行は `go run ./exercises/ex0X_xxx`、デバッグは該当 `main.go` を開いて `F5`。
> 各演習に「使ってほしいデバッグ機能」を書いています。print を足すのは封印して、その機能で解くのが練習です。

---

## 演習1 — `ex01_index`：添字の範囲外

スライスの末尾の要素を表示したいだけのプログラムですが、実行すると落ちます。

**やること**
- 実行して panic メッセージを読む。
- `lastValue` の中にブレークポイントを置き、`nums`、`len(nums)`、そして実際にアクセスしようとしている添字を観察する。
- なぜその添字が不正なのかを説明する。

**今回覚える機能**: ブレークポイントの設置 / 変数パネル / マウスホバーでの値確認 / ウォッチ式（`len(nums)` を登録してみる）

## 演習2 — `ex02_nilptr`：nil ポインタ参照

ユーザー一覧から名前で検索して、その人の情報を表示するプログラム。実行すると落ちます。

**やること**
- panic がどの行で出ているか確認する。
- その行にブレークポイントを置き、表示しようとしている変数（ポインタ）の中身を観察する。
- 「なぜそれが nil なのか」を、`findUser` の中に入って（ステップイン）追いかける。

**今回覚える機能**: ステップイン（`F11`）/ コールスタック / nil の見分け方 / 「症状の出る場所」と「原因の場所」が違うことの体感

## 演習3 — `ex03_conditional`：0 除算（条件付きブレークポイントの練習）

売上データから平均単価を計算して順に表示するプログラム。最初のいくつかは表示されますが、途中で落ちます。

**やること**
- 実行して、どこまで表示されてから落ちるかを見る。
- `averagePrice` にブレークポイントを置くと、ループのたびに止まって面倒です。**条件付きブレークポイント**を使い、問題が起きるケースだけで止める。
- 止まったら、その売上レコードの中身を観察して原因を説明する。

**今回覚える機能**: 条件付きブレークポイント（赤丸を右クリック →「ブレークポイントの編集」→ 条件式）

> ヒント：止めたい瞬間の条件を式で書きます。「割る数がどうなっていると割り算は壊れるか？」を考えてみてください。

## 演習4 — `ex04_callstack`：深い呼び出しでの範囲外（コールスタックの練習）

注文された商品名から価格表を引いて合計するプログラム。実行すると落ちます。

**やること**
- panic は `priceOf` の中で出ます。そこにブレークポイントを置き、価格表を引くのに使っている添字を観察する。
- その添字は別の関数（`indexOf`）が返した値です。**ステップイン**して `indexOf` の戻り値がどうなるかを追う。
- **コールスタック**を見て、「`main` のループの何回目・どの商品名」でこの事故が起きたのかを特定する。

**今回覚える機能**: ステップイン / ステップアウト（`Shift+F11`）/ コールスタックを上下に行き来して各階層の変数を見る

## 演習5 — `ex05_assertion`：型アサーション失敗

「数値が入っているはず」のリストを合計するプログラム。実行すると落ちます。

**やること**
- panic メッセージから、何を何に変換しようとして失敗したのかを読む。
- `sumNumbers` のループ内にブレークポイントを置き、各要素 `v` の**中身と動的な型**を観察する。
- どの要素が想定と違うのかを特定する。

**今回覚える機能**: ループ内ブレークポイント / 変数パネルでのインターフェース値（動的型）の確認 / デバッグコンソールで式を評価する

> ヒント：変数パネルで `v` を展開すると、入っている実際の型が見えます。条件付きブレークポイントを併用してもよいでしょう。

## 演習6 — `ex06_logpoint`：答えが間違っているだけのバグ（ログポイントの練習）

ここまでと違い、このプログラムは **panic しません**。実行は最後まで通りますが、**結果の数値が間違っています**。
開始残高に取引を順に適用した最終残高を計算するプログラムです。

**やること**
1. まず実行して、出てきた最終残高をメモする。手計算した正解と合わないことを確認する（`start=1000`、取引 `-200, 500, -100, -50, 300`）。
2. ループの中の代入行に **ログポイント** を仕掛ける。赤丸を右クリック →「**ログポイントの追加**」を選び、メッセージに `d={d}, balance={balance}` のように `{式}` を埋め込む。
3. `F5` で実行すると、**止まらずに**ループの各回の値がデバッグコンソールに流れる。print を一行も書かずに、毎回の `d` と `balance` の変化を追える。
4. ログを読んで、`balance` が「足し込まれていない」ことに気づく。どの行が原因かを説明する。

**今回覚える機能**: ログポイント（`fmt.Println` を仕込む代わり。コードを汚さない・消し忘れない）

**ついでに試すテクニック**（同じループで練習できます）
- **ヒット回数の条件**：ブレークポイントの編集で「ヒット回数」を `>3` などにし、ループ後半だけで止める。
- **変数の書き換え (Set Value)**：停止中に変数パネルで `balance` を右クリックして値を書き換え、その値で続行してみる。
- **カーソル位置まで実行 (Run to Cursor)**：停止中に別の行を右クリック →「カーソルまで実行」。
- **テストのデバッグ**：このフォルダには失敗する `balance_test.go` があります。テスト関数の上に出る「**debug test**」リンクを押すと、そのテストだけをブレークポイント付きで走らせられます。

## 演習7 — `ex07_deadlock`：goroutine デッドロック

複数の goroutine で値を計算し、チャネルで集めて合計するプログラム。実行すると**固まって、最後にデッドロックのエラーで落ちます**。

**やること**
1. 実行して、出てくる `fatal error: all goroutines are asleep - deadlock!` と、その下の **goroutine 一覧（ダンプ）** を読む。どの goroutine が、どの行で、何待ち（`chan receive` など）で止まっているかを読み取る。
2. main がチャネルから受信している回数と、goroutine が送信している個数を数えて見比べる。
3. なぜ全 goroutine が眠ってしまうのかを説明する。

**今回覚える機能**: デッドロックのダンプの読み方 / コールスタックパネルでの **goroutine の切り替え** / 固まった（自動検出されない）ハング時の **一時停止 (Pause)** で各 goroutine の停止位置を見る

> ヒント：「いくつ送られて、いくつ受け取ろうとしているか」。数が合わないと、足りない分の受信は永遠に相手を待ち続けます。
