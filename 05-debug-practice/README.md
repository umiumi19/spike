# go-debug-practice

VS Code のブレークポイントを使ったデバッグを身につけるための、小さな練習用リポジトリです。
`fmt.Println` を仕込んで変数を覗くやり方から卒業して、**実行を止めて変数・コールスタックをその場で観察する**やり方に慣れることを目的にしています。

各演習は **コンパイルは通るのに実行時に panic する**コードです。
panic メッセージだけ見ても「どこで」起きたかは分かりますが、「**なぜ**その値になったのか」を突き止めるのがデバッグの本番です。そこをブレークポイントで追う練習をします。

---

## 1. 必要なもの

- **Go**（1.21 以上であれば何でも可。このリポジトリは 1.22 で検証済み）
  - 確認: `go version`
- **VS Code**
- **Go 拡張機能**（拡張機能パネルで `golang.go` を検索してインストール）
- **Delve**（Go 用デバッガ。VS Code が初回デバッグ時に自動でインストールを促します。手動なら `go install github.com/go-delve/delve/cmd/dlv@latest`）

> Go 拡張を入れた直後は、コマンドパレット（`Cmd/Ctrl+Shift+P`）→ `Go: Install/Update Tools` で `dlv` にチェックを入れて入れておくと確実です。

---

## 2. 開き方

このフォルダ（`go-debug-practice`）を **そのまま VS Code で開いてください**（`File > Open Folder`）。
`.vscode/launch.json` が同梱されているので、デバッグ構成はすぐ使えます。

---

## 3. ブレークポイントデバッグの最短ガイド

print デバッグとの一番の違いは、**コードを書き換えずに、止めて、覗いて、また進める**という点です。

### ブレークポイントを置く
- 行番号の**左側の余白**をクリックすると赤い丸が付きます（= ブレークポイント）。`F9` でも切り替え可。
- そこにプログラムが来ると**実行が一時停止**します。

### デバッグを開始する
- 止めたいコードのある `main.go` をエディタで開く。
- `F5` を押す（または左の「実行とデバッグ」ビュー上部のドロップダウンで構成を選んで再生ボタン）。
- 構成 **「▶ 今開いている演習をデバッグ」** は、いま開いているファイルの演習をそのまま実行します。`ex01 index` などを選べば個別指定も可能です。

### 止まったら見る場所（左の「実行とデバッグ」ビュー）
| パネル | 何が見えるか |
|---|---|
| **変数 (Variables)** | 今のスコープのローカル変数・引数・レシーバ。構造体は ▸ で展開できる |
| **ウォッチ式 (Watch)** | `len(nums)` のような任意の式を登録して常時評価 |
| **コールスタック (Call Stack)** | 今ここに辿り着くまでの関数の呼び出し履歴。クリックでその階層の変数に切り替わる |
| **デバッグコンソール (Debug Console)** | その場で式を打って評価できる（`nums[2]` など） |

変数の値は、ソースコード中の変数名に**マウスを乗せる**だけでもポップアップで見られます。

### 進め方（停止中に使うツールバー）
| 操作 | キー | 意味 |
|---|---|---|
| 続行 Continue | `F5` | 次のブレークポイントまで一気に進む |
| ステップオーバー Step Over | `F10` | 現在行を実行。関数呼び出しは中に入らず一気に飛び越える |
| ステップイン Step Into | `F11` | 現在行の関数呼び出しの**中へ**入る |
| ステップアウト Step Out | `Shift+F11` | 今の関数を最後まで実行して**呼び出し元へ**戻る |
| 再起動 / 停止 | | 最初からやり直す / デバッグ終了 |

### 条件付きブレークポイント（演習3で使います）
- ブレークポイントの赤丸を**右クリック → ブレークポイントの編集**。
- 「式」に `s.Quantity == 0` のような条件を書くと、**その条件が真のときだけ**止まります。ループの特定回だけ捕まえたいときに強力です。

### ログポイント（演習6で使います／print デバッグの置き換え）
- 赤丸を**右クリック →「ログポイントの追加」**。メッセージに `idx={idx}, name={name}` のように `{式}` を埋め込むと、**実行を止めずに**その行を通るたびにログだけ出します。
- `fmt.Println` を仕込んで・消して・消し忘れる、という手間がなくなります。コードを一切汚さずに print 相当のことができる、と考えてください。

### panic で自動的に止める設定（おすすめ）
「実行とデバッグ」ビューの下のほうにある **「ブレークポイント」** セクションで、**`Uncaught Exceptions`（補足されない例外）にチェック**を入れておくと、panic が起きた**まさにその瞬間**に自動で停止し、その時点の変数とコールスタックを観察できます。原因究明が一気に楽になります。

---

## 4. 進め方

1. `docs/exercises.md` を開き、演習1から順に取り組む。
2. まずは `go run ./exercises/ex01_index` のように**普通に実行**して、どんな panic が出るか自分の目で見る。
3. 次に、メッセージに出ている行のあたりにブレークポイントを置いて `F5` でデバッグ実行し、変数を観察して**根本原因**を突き止める。
4. 詰まったら `docs/solutions.md` を見る。**答えだけでなく、ブレークポイントを使ってそこへ辿り着く手順**を載せています。

まずは自力で1〜2分粘ってみて、それから解答を見るのがおすすめです。

```

├── .vscode/
│   └── launch.json          ← デバッグ構成（編集不要）
├── exercises/
│   ├── ex01_index/main.go   ← 添字の範囲外
│   ├── ex02_nilptr/main.go  ← nil ポインタ参照
│   ├── ex03_conditional/main.go ← 0 除算（条件付きBPの練習）
│   ├── ex04_callstack/main.go   ← 深い呼び出しでの範囲外（コールスタックの練習）
│   ├── ex05_assertion/main.go   ← 型アサーション失敗
│   ├── ex06_logpoint/           ← 答えが違うだけのバグ（ログポイント＋テストのデバッグ）
│   │   ├── main.go
│   │   └── balance_test.go
│   └── ex07_deadlock/main.go    ← goroutine デッドロック（goroutine の観察）
└── docs/
    ├── exercises.md         ← 問題（やること）
    └── solutions.md         ← 解答＋デバッグ手順
```
