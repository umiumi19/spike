# 技術選定スパイク・リポジトリ(骨組み)

迷った4要素を「手で動かして確かめる」ための最小スパイク集。
ロジックの肝は `TODO` にしてあります。自分で埋めて書き心地・つながりを体感するのが目的です。

## 目的の枠(重要)

- これは「再検討」ではなく **確認・学習**。選んだ道が動くか/手に馴染むかを見る。
- **時間を区切る**(目安: 週末1〜2日)。全部を作り込んで比較沼にはまらない。

## 前提ツール

- Go 1.22+
- Docker(ローカルPostgres用)
- Flutter SDK(2のDart生成のみ)
- 各CLIは各スパイクのREADMEでインストール

## DB構成

各スパイクは独立したDB構成を持つ。共通スキーマは存在しない。

| スパイク            | DB                                                 | 起動方法                                  |
| ------------------- | -------------------------------------------------- | ----------------------------------------- |
| `01-db-access/`     | `blog_gorm` / `blog_ent` / `blog_sqlc`(別コンテナ) | `cd 01-db-access && docker compose up -d` |
| `02-api-toolchain/` | なし(インメモリ)                                   | —                                         |
| `03-router/`        | なし(インメモリ)                                   | —                                         |
| `04-migration/`     | `spike`(ルートのコンテナ)                          | `docker compose up -d`                    |

## 4つのスパイク

| #   | フォルダ            | 何を確かめる                                               | 性質               |
| --- | ------------------- | ---------------------------------------------------------- | ------------------ |
| 1   | `01-db-access/`     | ent / go-jet / sqlc で**同じ複雑クエリ**を書き比べ         | 書き心地が決め手   |
| 2   | `02-api-toolchain/` | net/http+Huma → OpenAPI → Dart生成が**最後までつながるか** | つながり検証       |
| 3   | `03-router/`        | net/http ⇔ chi ⇔ Echo の**差し替えが数行か**               | 差し替えコスト確認 |
| 4   | `04-migration/`     | Atlas / goose / golang-migrate の**運用感**                | 手触り比較         |

## おすすめの順番

1. **4(マイグレーション)** — `docker compose up -d` → Atlas で運用感を確認
2. **1(DBアクセス)** — `cd 01-db-access && docker compose up -d` → 同じクエリを3通り書き比べ ← 本命
3. **2(APIツールチェーン)** — DB不要。一気通貫を1本通す
4. **3(ルーター差し替え)** — DB不要。最後に軽く確認

## go.mod について

ルートに最小の `go.mod` を置いてある。各スパイクで使うライブラリは
`go get` で都度追加し、`go mod tidy` で整える(各READMEに記載)。
